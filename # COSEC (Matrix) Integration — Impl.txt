# COSEC (Matrix) Integration — Implementation Plan for **Ofis Square**

**Version:** 1.0

**Purpose:** This document provides a complete, actionable implementation plan to integrate COSEC (Matrix) access control into the Ofis Square backend and frontend. It covers architecture, DB schemas, environment configuration, low-level API wrapper, business services, controllers & routes, hooks into existing flows (member onboarding, contracts, day-pass, visitors), cron jobs, frontend pages, testing, security, operational runbook, and sample requests.

---

## Executive summary

Ofis Square will use the COSEC Web API (Matrix) to provision users, assign credentials (RFID/cards), grant/revoke access to doors/zones/groups, and fetch access events. Integration will follow a layered approach:

* `utils/matrixApi.js` — low-level HTTP client with Basic Auth.
* `services/matrixService.js` — business orchestration (provision, assign, revoke, reconciliation).
* `models/*` — DB mappings for matrix users and building/device metadata.
* `controllers/matrixController.js` + `routes/matrix.js` — admin endpoints for manual operations.
* Hooks in existing controllers (`memberController`, `contractController`, `dayPassController`, `visitorController`) for automatic provisioning and revocation.
* Cron jobs: expiry watcher, reconciliation, event poller.
* Frontend: admin UI pages for Matrix Users, Building Access Setup, Access Logs.

Key design constraints:

* Use Basic HTTP Authentication with credentials stored in `.env`.
* Idempotent operations and retry/backoff for reliability.
* Activity logging for every operation.
* Admin-only endpoints for manual control.

## Prerequisites & assumptions

* You have access to a COSEC (Matrix) server and network reachability from backend. If COSEC is behind a firewall, ensure server IP is whitelisted.

* Ofis Square backend uses Node.js/Express and MongoDB; integrate using Mongoose models.

## Database schemas (Mongoose models)

### 1. `matrixUserModel.js`

Fields:

* `_id` (ObjectId)
* `userId` — reference to local user/member/onDemandUser/visitor
* `userType` — enum: `MEMBER | DAY_PASS | VISITOR | ADMIN` (helps policies)
* `matrixUserId` — id returned/used by COSEC (string)
* `credentials` — array of objects `{ type: 'card'|'fp'|'face', data: String, facilityCode?: String, issuedAt?: Date }`
* `assignedDevices` — array `{ deviceId, deviceName, assignedAt, schedule?: { start, end, daysOfWeek, startTime, endTime } }`
* `provisionedAt`, `lastSyncAt`, `revokeAt` (Date nullable), `status` (`active|revoked|pending`)
* `meta` — free-form JSON for COSEC response codes

### 2. `buildingMatrixModel.js` (or extend `buildingModel`)

Fields:

* `buildingId` — ref to building
* `matrixSiteId` — COSEC site or location id
* `deviceMap` — array of `{ localDoorName, deviceId (COSEC), deviceCode, deviceType, panelId? }`
* `defaultAccessRoutes` — map role -> device list or group id

### Indexes & considerations

* Index on `userId` and `matrixUserId` for quick lookups.
* TTL or scheduled jobs will manage `revokeAt` values.

## Low-level API wrapper — `utils/matrixApi.js`

Responsibilities:

* Centralize Basic Auth header generation.
* Provide methods for COSEC endpoints (create user, set-credential, assign/revoke device, fetch events, get devices).
* Normalize responses to JSON.
* Implement retries + exponential backoff and circuit-breaker behavior.

### Sample implementation (concept)

```js
// ofis-square/utils/matrixApi.js
import axios from 'axios';
import { Buffer } from 'buffer';

const COSEC_BASE = process.env.COSEC_API_BASE;
const AUTH_HEADER = 'Basic ' + Buffer.from(`${process.env.COSEC_API_USER}:${process.env.COSEC_API_PASS}`).toString('base64');

const api = axios.create({ baseURL: COSEC_BASE, timeout: parseInt(process.env.COSEC_API_TIMEOUT_MS || 10000), headers: { Authorization: AUTH_HEADER, 'Content-Type': 'application/json' } });

export const matrixApi = {
  async getDevices() { return api.get('/device?action=get').then(r => r.data); },
  async createUser(payload) { return api.post('/user?action=set', payload).then(r => r.data); },
  async deleteUser(id) { return api.post('/user?action=del', { id }).then(r => r.data); },
  async setCredential({ id, type, data, facilityCode }) { return api.post('/user?action=set-credential', { id, credentialType: type, data, facilityCode }).then(r => r.data); },
  async delCredential({ id, type }) { return api.post('/user?action=del-credential', { id, credentialType: type }).then(r => r.data); },
  async assign({ id, devices }) { return api.post('/device?action=assign', { id, device: devices.join(',') }).then(r => r.data); },
  async revoke({ id, devices }) { return api.post('/device?action=revoke', { id, device: devices.length ? devices.join(',') : 'all' }).then(r => r.data); },
  async getEvents(params) { return api.get('/event-acs?action=get', { params }).then(r => r.data); }
};
```

**Notes:**

* Some COSEC APIs expect query-string parameters and may also accept `format=json`; test and adapt.
* Keep `requestId` or correlation id in headers for tracing.

## Business orchestration — `services/matrixService.js`

Responsibilities:

* High-level actions (provisionMember, assignCard, grantAccessForContract, revokeOnExpiry, grantDayPassAccess, revokeDayPassAccess).
* Handle idempotency: before creating user, check `matrixUserModel` for existing `matrixUserId`.
* Persist matrix responses and log to `activityLogModel`.
* Schedule or set `revokeAt` for temporary access and write to DB.

### Key functions

* `provisionForMember(member, contract)`

  * If `matrixUserModel` exists: update COSEC user (name/email)
  * Else: call `matrixApi.createUser`, persist `matrixUserId`.
  * If member has an RFID/card: call `matrixApi.setCredential` and store credential.
  * Assign access: determine devices from `buildingMatrixModel` and `contract` windows and call `matrixApi.assign`.
  * Return consolidated status and log it.

* `revokeOnContractExpiry(member)`

  * Call `matrixApi.revoke({ id: matrixUserId, devices: [] })` to revoke all
  * Optionally call `matrixApi.delCredential` if you issue and want to remove cards
  * Update `matrixUserModel.status = 'revoked'` and set `revokeAt`.

* `grantDayPassAccess(dayPass)`

  * Create or reuse matrixUser, assign devices with schedule for single date
  * Set `revokeAt` to end-of-day

* `pollAndIngestEvents(since)`

  * Call `matrixApi.getEvents({ from: since, to: now })` and bulk insert events into `activityLogModel`.

**Implementation tip:** Return detailed error objects so calling controllers can choose to retry or create alerts.

## Controllers & routes

Add `controllers/matrixController.js` with admin-protected actions, and register `routes/matrix.js` in `routes/api.js`.

Endpoints suggested (all admin guarded):

* `POST /api/matrix/provision-user` — body: { userId, userType, name, phone, email }
* `POST /api/matrix/set-credential` — body: { userId, type, data, facilityCode }
* `POST /api/matrix/assign` — body: { userId, devices: [] }
* `POST /api/matrix/revoke` — body: { userId, devices: [] }
* `GET  /api/matrix/devices` — list devices
* `GET  /api/matrix/events` — query events

**Controller responsibilities:**

* Validate request & permissions
* Call `matrixService` and log results
* Return human-friendly responses (not raw COSEC payloads), but include raw response in detailed logs for debugging

## Hooks in existing flows

Integrate matrix actions into these places:

1. **Member onboarding (memberController)**

   * On create and when assigned to a building with active contract, call `matrixService.provisionForMember` and `matrixService.grantAccessForContract`.
   * Set matrix mapping (`matrixUserModel`) and store `matrixUserId` in member profile if helpful.

2. **Contract activation/expiry (contractController)**

   * On activation: ensure members attached to contract get required access.
   * On expiry: call `matrixService.revokeOnContractExpiry`.

3. **Day pass (dayPassController)**

   * On successful payment/booking: call `matrixService.grantDayPassAccess` and schedule revoke at end-of-day.

4. **Visitors (visitorController)**

   * When invite accepted: create visitor matrix user and grant access for specified window; schedule revoke.

**Important:** Wrap matrix calls in `try/catch` and ensure the main user flow (e.g., onboarding response) never fails completely if matrix is temporarily down — return success to user and enqueue a retry for provisioning, but **log** the failure and alert admins.

## Cron jobs & background tasks

1. **Expiry watcher (every 5–15 minutes)**

   * Query `matrixUserModel` for `revokeAt <= now` and status `active`.
   * Call revoke, update status.

2. **Reconciliation job (nightly)**

   * For each building, fetch COSEC `device?action=get` and compare assignments vs local DB.
   * Create discrepancy report and optionally auto-fix or alert.

3. **Event poller (every 15–60 seconds depending on load)**

   * Use `event-acs?action=get` or `event-acs-date` to fetch events since last timestamp and persist them.

Implement jobs using your existing cron mechanism or a job queue (Bull/Agenda) if available.

## Error handling & idempotency

* Check existence before creating: if the COSEC returns an error about duplicate user, fetch and store `matrixUserId`.
* Use local DB `matrixUserModel` as source-of-truth for provisioning status; keep COSEC as the system of action.
* Retriable failures: transient network errors or 5xx should be retried with exponential backoff up to `COSEC_API_MAX_RETRIES`.
* Non-retriable: malformed requests or auth errors should fail fast and alert admins.

---

## Security

* Never commit `COSEC_API_USER` / `COSEC_API_PASS` to source control.
* Store secrets in environment or secrets manager.
* Limit admin endpoints via existing RBAC middleware.
* Log COSEC responses but redact sensitive credential material (e.g., card numbers) in logs — store hashed or truncated versions.

## Operational runbook (short)

**If COSEC unreachable:**

1. Check backend server network and DNS to COSEC host.
2. Check `.env` credentials and perform a curl test from backend host.
3. Check COSEC service status and logs.
4. Reconcile missed provisioning by running `reconciliation` job.

**If wrong assignments detected:**

1. Run `GET /device?action=get` to fetch live state.
2. Compare with local DB; either auto-fix or generate alert for manual resolution.

**Audit:** Always include COSEC response codes in Activity Logs for postmortem.

### matrixService.provisionForMember (simplified)

```js
import { matrixApi } from '../utils/matrixApi';
import MatrixUser from '../models/matrixUserModel';
import BuildingMatrix from '../models/buildingMatrixModel';

export async function provisionForMember(member, contract) {
  let m = await MatrixUser.findOne({ userId: member._id });
  if (!m) {
    const res = await matrixApi.createUser({ id: `m_${member._id}`, name: `${member.firstName} ${member.lastName}` });
    m = await MatrixUser.create({ userId: member._id, matrixUserId: res.id, provisionedAt: new Date(), status: 'active' });
  }

  // assign devices based on building mapping
  const building = await BuildingMatrix.findOne({ buildingId: member.buildingId });
  const devices = building.defaultAccessRoutes.MEMBER || building.deviceMap.map(d => d.deviceId);
  await matrixApi.assign({ id: m.matrixUserId, devices });
  m.assignedDevices = devices.map(d => ({ deviceId: d, assignedAt: new Date() }));
  await m.save();
}
```
